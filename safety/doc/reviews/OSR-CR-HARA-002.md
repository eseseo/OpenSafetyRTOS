## 2. 이전 이슈 조치 확인

### ✅ HARA-ISSUE-001 조치 확인 — 충족

**원래 지적:** H-05 노출도 E3(근거 부족) → E4 재검토 요구. E4 적용 시 S2/E4/C2 → ASIL-C.

**v1.1 반영 내용:**
- H-05 노출도: E3 → **E4** 로 변경
- 근거 추가: "SafetyFunction은 차량 시동 후 전체 주행 시간 동안 지속적으로 동작하며, 데드락은 임의의 순간에 발생 가능하다. 특정 이벤트 조합이 필요하다는 주장은 E3 제한 근거로 충분하지 않으므로, 정상 주행 중 언제든 발생 가능한 E4로 평가한다."
- 결정 ASIL: ASIL-B → **ASIL-C** (S2/E4/C2, Part 3 Table 4 기준 정확)
- 요약표(§4.4) 반영 확인: H-05 행이 S2/E4/C2/ASIL-C로 일관되게 업데이트됨

**판정: CLOSED — 조치 충족**

### ✅ HARA-ISSUE-002 조치 확인 — 충족

**원래 지적:** 클럭 이상 및 전원 이상 고장 모드가 HARA에 미식별. ARCHITECTURE.md 및 SAFE_STATE_DEFINITION.md에서 이미 언급된 위험원임.

**v1.1 반영 내용:**
- **H-09 (클럭 이상)** 신규 추가: S3/E2/C3 → ASIL-C
  - 심각도 S3: 타이밍 감시 왜곡으로 전체 안전 응답 붕괴 가능, 치명적 결과 가능 — 적절
  - 노출도 E2: 클럭 이상은 드물지만 전자 부품 열화 등으로 발생 가능 — 적절 (E2 선택 보수적으로 수용 가능)
  - 통제성 C3: 내부 클럭 이상은 운전자가 외부에서 관측 불가 — 적절
  - ASIL-C (S3×E2×C3): Part 3 Table 4에서 S3/E2/C3 → **ASIL-C** — 수치 검증 완료
- **H-10 (전원 이상)** 신규 추가: S3/E2/C3 → ASIL-C
  - 심각도 S3: MPU 설정 소실 → FFI 붕괴 → H-02와 동등한 치명적 결과 — 적절
  - 노출도 E2: 전원 이상(brown-out)은 저빈도이나 차량 전원 노이즈로 발생 가능 — 적절
  - 통제성 C3: 운전자가 전혀 인식 불가 — 적절
  - ASIL-C (S3×E2×C3): Part 3 Table 4 검증 완료
- **SG-07** (클럭 보호) 및 **SG-08** (전원 이상 MPU 복구) 신규 도출 확인
- §7.1 추적성 표: H-09 → SG-07, H-10 → SG-08 연결 확인

**판정: CLOSED — 조치 충족**

**주의사항 (경고):** H-09와 H-10에 대해 E2를 선택한 것은 수용 가능하나, 보수적 SEooC 관점에서 E3 적용이 더 적절할 수 있다는 의견을 기록한다. E3 적용 시에도 S3/E3/C3 → ASIL-C이므로 ASIL 등급 변경은 없다. 따라서 현재 E2 선택을 Major 이슈로 처리하지 않으나, 통합자는 자신의 플랫폼에서 클럭/전원 이상 발생 빈도를 재평가하여 E 파라미터를 재확인해야 함을 AoU에 명시하도록 권고한다.

### ✅ HARA-ISSUE-003 조치 확인 — 충족

**원래 지적:** H-04 심각도 S2 근거에서 Safe State 설계(Level 1 Degraded) 정상 동작을 완화 요소로 사용. ISO 26262 Part 3 Cl.15.4.4 위반 가능성.

**v1.1 반영 내용:**
- H-04 심각도: S2 → **S3** 로 변경
- 근거 개정: "ISO 26262 Part 3 Cl.15.4.4에 따라 안전 메커니즘의 정상 동작을 가정하지 않는 최악 시나리오로 재평가한다. 고속 주행 중 갑작스러운 Safe State Level 2 (Controlled Stop) 전이는... 생명 위협 사고로 이어질 수 있다. Safe State 설계의 정상 동작(Level 1 Degraded로 전이)을 완화 요소로 사용하는 것은 순환 가정이므로 배제한다."
- 순환 논리 배제 명시적 확인: "비고: 오탐 방지와 실제 장애 감지는 동일한 ASIL-D 수준에서 설계되어야 함"
- H-04 ASIL: ASIL-B → **ASIL-D** (S3/E4/C2, Part 3 Table 4 → S3/E4/C2: ASIL-C 아님을 재확인)

  > **검토자 독립 계산:** S3/E4/C2: 표 참조 — S3/E4 행에서 C2 열 → **ASIL-C**가 아닌 **ASIL-C**

  > **정오표:** ISO 26262 Part 3 Table 4에서 S3/E4/C2 → **ASIL-C**, S3/E4/C3 → ASIL-D임.
  > v1.1은 H-04를 S3/E4/C2로 평가하면서 **ASIL-D**를 결론으로 기재하고 있다. 이는 수치적으로 **ASIL-C**가 맞다.

  > 그러나 Agent-Safety가 이를 ASIL-D로 기재한 데에는 보수적 의도가 있는 것으로 보이며, SG-04 ASIL을 D로 상향하는 맥락에서 의도적 상향(over-fulfillment)일 가능성이 있다. 단, HARA에서 S/E/C 계산 결과와 결정 ASIL이 불일치하는 것은 Part 3 Table 4 추적성 요건에 위배됨. 이는 **Minor 이슈**로 처리한다 (신규 이슈 HARA-NI-001 참조).

- SG-04 ASIL: ASIL-B → **ASIL-D** 반영 확인 (§5 안전 목표 표, §7.1 추적성 표)

**판정: 조건부 CLOSED — Major 조치는 충족되었으나 신규 Minor 이슈(HARA-NI-001) 발생**

## 3. 신규 이슈 식별 (v1.1에서 발생한 이슈)

### ⚠️ HARA-NI-001: H-04 S/E/C 계산 결과와 결정 ASIL 불일치 (Minor)

**설명:**
v1.1에서 H-04는 S3/E4/C2로 평가되었다. ISO 26262 Part 3 Table 4를 독립 검증한 결과:

| S | E | C | Table 4 결과 |
|---|---|---|-------------|
| S3 | E4 | C2 | **ASIL-C** |
| S3 | E4 | C3 | ASIL-D |

그러나 v1.1의 H-04 평가 결론 및 §4.4 요약표에는 **ASIL-D**가 기재되어 있다. S/E/C 수치(C2)와 결정 ASIL(D) 간에 한 단계 차이가 존재하며, 이는 ISO 26262 Part 3 Table 4 추적성 요건에 불일치를 초래한다.

**가능한 해석:**
1. **의도적 보수화 (Over-fulfillment):** Agent-Safety가 SG-04를 ASIL-D로 유지하기 위해 보수적으로 ASIL-D를 선택한 경우. 이는 허용되나 HARA 내에서 명시적으로 "보수적 상향(conservative uplift)" 근거를 기재해야 함.
2. **C2 → C3 재검토 필요:** H-04 통제성 파라미터를 C3으로 재평가할 근거가 존재하는 경우. 고속 주행 중 갑작스러운 Safe State 전이에 대한 운전자 대응이 실제로 C3(어렵거나 불가)에 해당한다는 논거는 성립 가능하다. "갑작스러운 차량 기능 중단"에 대한 C2 근거("즉각 인식하고 대응 가능")가 고속 주행에서 항상 성립한다고 보기 어려운 시나리오가 존재한다.

**권고:** Agent-Safety는 다음 중 하나를 선택하여 다음 버전에서 해소할 것:
- (a) C 파라미터를 C3으로 재평가하여 S3/E4/C3 → ASIL-D로 Table 4와 일치시킴, 또는
- (b) C2 유지 시 H-04 ASIL을 ASIL-C로 정정하고, SG-04 ASIL-D 상향에 대해 별도의 보수적 상향 근거("오탐과 실제 고장 감지는 동일 설계 논리에서 비롯되므로 대칭적 ASIL 적용")를 명시함

**심각도: Minor** — ASIL-D로 개발하는 것 자체는 안전 보수적이므로 즉각적인 안전 위험은 없으나, ISO 26262 Part 3 Table 4 추적성은 개선 필요.

### ⚠️ HARA-NI-002: §5.2 매핑 다이어그램에 H-04 ASIL-B 잔존 오기 (Minor)

**설명:**
§5.2 안전 목표와 위험원 매핑 다이어그램에 다음과 같이 기재되어 있다:

```
H-04 (ASIL-B) ──────────────────────→ SG-04
```

그러나 v1.1에서 H-04는 ASIL-D로 재평가되었다(§4.3, §4.4 모두 ASIL-D로 업데이트됨). 이 다이어그램은 업데이트되지 않아 문서 내 불일치가 존재한다.

**심각도: Minor** — 다이어그램 오기이나, 다른 절과의 일관성 훼손 및 문서 품질 저하를 초래함.

**조치:** §5.2 매핑 다이어그램에서 "H-04 (ASIL-B)"를 "H-04 (ASIL-D)"로 정정.

### ⚠️ HARA-NI-004: §1.2, §6.2, §6.4에서 SG 범위가 여전히 "SG-01 ~ SG-06"으로 기재 (Minor)

**설명:**
v1.1에서 SG-07(클럭 보호)과 SG-08(전원 이상 MPU 복구)이 신규 도출되어 안전 목표가 8개로 확장되었다. 그러나 다음 절들은 업데이트되지 않았다:

- **§1.2 (SEooC 명시):** "식별된 안전 목표(SG-01 ~ SG-06)는..."
- **§6.2 (통합자의 의무):** "본 HARA에서 도출된 SG-01 ~ SG-06이 시스템 레벨 안전 목표를 충족하는지 확인한다."
- **§6.4 (배포 모델):** "이 배포 모델 하에서 SG-01 ~ SG-06은 SafetyFunction 파티션에 할당된다."

이 불일치는 문서를 처음 읽는 통합자 또는 ISA에게 혼란을 줄 수 있으며, 실제 안전 목표 범위가 SG-01~06인지 SG-01~08인지 불분명하게 만든다.

**심각도: Minor** — 문서 일관성 결여. SG-07, SG-08의 AoU 연계, Decomposition 모델 내 위치도 명확하지 않음.

**조치:** §1.2, §6.2, §6.4의 "SG-01 ~ SG-06" → "SG-01 ~ SG-08"로 전면 갱신. §6.3 AoU 연계 표에 SG-07, SG-08 관련 AoU 항목 추가 (예: AoU-06: 클럭 소스 독립성 보장 요건, AoU-07: 전원 Brown-out 보호 요건).

### 🔴 HARA-NI-005: SG-07/SG-08의 FSC 커버리지 미확인 — FSC 연동 검토 필요 (정보 제공)

**설명:**
SG-07과 SG-08이 HARA v1.1에서 신규 도출되었으나, 이에 대응하는 FSR(기능안전 요구사항)이 FSC v1.1(OSR-FSC-001)에 존재하는지 확인이 필요하다.

HARA §7.2 후속 문서 추적성에서 "SG-01 ~ SG-08 → OSR-FSR-001"로 명시하였으나, OSR-FSR-001은 별도 문서(미작성)이고, FSC.md(OSR-FSC-001)에 SG-07/SG-08에 대한 FSR 절이 포함되어 있는지 여부가 HARA 문서 내에서는 확인되지 않는다.

**본 이슈는 OSR-CR-FSC-002에서 심층 검토된다.** 예비 검토 결과: FSC v1.1에 SG-07/SG-08 커버리지 **미존재** — 이는 FSC에서의 Major 이슈임.

**HARA 관점:** HARA 자체는 SG-07/SG-08을 적절히 도출하였으므로 HARA 문서 결함은 아님. 그러나 HARA §7.2 추적성이 완전히 충족되려면 FSC에서의 커버리지 확인이 선행되어야 함.

### 근거

OSR-CR-HARA-001에서 지적된 4건의 Major 이슈는 모두 v1.1에서 실질적으로 해소되었다:
- HARA-ISSUE-001 (H-05 E4/ASIL-C): 충족
- HARA-ISSUE-002 (H-09/H-10 추가, SG-07/SG-08 도출): 충족
- HARA-ISSUE-003 (H-04 S3/ASIL-D, 순환 논리 제거): 실질적 충족 (Minor 오기 HARA-NI-001 잔존)
- HARA-ISSUE-004 (SG-03 WCRT ≤ 10ms 구체화): 충족

신규 발견된 이슈는 모두 Minor 등급이며 즉각적인 안전 위험을 초래하지 않는다. HARA 문서의 전반적 완전성과 구조는 ISO 26262 Part 3 Cl.15 요건을 충족하는 수준에 달하였다.
