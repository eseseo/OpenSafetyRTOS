## 검토 결과 요약

**판정: 조건부 승인 (CONDITIONAL PASS)**

주요 강점: HARA 구조가 ISO 26262 Part 3 Cl.15를 전반적으로 충족하며, 8개 위험원 식별과 SEooC 경계 처리가 양호함. ASIL 결정 매트릭스(Part 3 Table 4) 인용이 명시적으로 포함되어 있음.

주요 우려: H-05(Watchdog kick 실패)의 ASIL-B 등급이 재검토 요구됨. 클럭/전원 고장 모드 위험원 미식별. H-04의 심각도(S2) 근거에 차량 유형 편향 존재. SG-03의 [X]ms 파라미터 관리 방식이 기능 요구사항으로서 불완전.

---

## 항목별 검토

### ✅ 적합 항목

1. **SEooC 경계 명시 (ISO 26262-8 Cl.13 준수):** §1.2에서 SEooC HARA임을 명확히 선언하고, 실제 차량 기능을 알 수 없는 상황에서의 분석 한계 및 통합자의 의무를 명시하였다. AoU(Assumption of Use) 참조 문서(OSR-AOU-001)와의 연계가 §6.3에서 구체적으로 기술되어 있음.

2. **운영 상황의 다양성 (OS-01~05):** 정상 주행, 시동, 주차/저속, 긴급 상황, 통신 오류 등 5개 운영 상황은 일반적인 차량용 ECU에 적용 가능하며, 각각에 노출도 근거가 제시되어 있음. 통신 오류 상황(OS-05)을 별도로 정의한 것은 Mailbox 기반 FFI 아키텍처에서 적절한 식별임.

3. **ASIL 결정 매트릭스 명시 (ISO 26262 Part 3 Table 4):** §4.2에서 S/E/C 조합별 ASIL 결정 매트릭스를 문서 내에 완전하게 재현하였으며, 각 위험원에 대해 이 매트릭스를 명시적으로 적용하여 추적 가능성을 확보하였음.

4. **위험원-안전목표 추적성 (§7):** H-01~H-08 각각이 SG로 추적되며, 미연결 위험원(H-05)의 경우 SG-01 FSR로 처리됨을 명시하였음. 안전목표 → 후속 문서(FSR, FMEA, AoU, SSR)로의 전향 추적성도 제시됨.

5. **DMA 위협 포함 (H-02, SG-02):** ARCHITECTURE.md §5.2의 확장 FFI 분석에서 식별된 DMA 경유 메모리 침범 위협이 H-02 위험원의 설명에 포함되어 있으며, FSR-02-04로 연결됨.

6. **인터럽트 벡터 테이블(IVT) FFI 경로 반영:** ARCHITECTURE.md §5.2에서 식별된 NVIC 벡터 테이블 덮어쓰기 위협이 H-02 범위 내에 포함되며, FSR-02-02로 전개됨.

7. **SG-03 파라미터 관리 (§5.1):** 미결 파라미터 [X]ms에 대해 참조값 10ms를 제시하고 통합자 AoU-03으로 결정 책임을 위임하는 구조를 명시하였음.

### ⚠️ 지적사항 (Issues)

**ISSUE-001: H-05 ASIL-B 등급 재검토 필요 — 과소평가 가능성**

- **설명:** H-05(Watchdog kick 실패 — SafetyFunction 데드락)에 대해 E3, S2, C2 → ASIL-B를 결정하였으나, 이 평가에 내재된 가정이 충분히 검증되지 않음.
  - 노출도 E3 근거: "특정 이벤트 조합이 필요"라고 서술하나, SafetyFunction은 차량 시동 후 전체 시간 동안 동작하며 데드락 조건은 임의의 순간에 발생 가능함. 이를 E3(중간)으로 제한할 근거가 약함. E4가 더 보수적으로 적절할 수 있음.
  - E4로 변경 시: S2/E4/C2 → **ASIL-C** (Part 3 Table 4 기준). 현재 ASIL-B보다 한 등급 높아짐.
  - 심각도 S2 근거: "MCU 리셋 후 정상 부팅이 되면 복구되는 시나리오"를 S2의 근거로 사용하나, 이는 회복 메커니즘의 정상 작동을 전제함. 회복이 실패하거나 리셋 후 부팅 자가진단이 결함을 감지하지 못하는 복합 시나리오(H-05 + H-07 연쇄)에서는 S3에 해당할 수 있음.
  - H-05는 SG-01에 부분 포괄됨으로 표기되나, SG-01은 ASIL-D임. H-05의 ASIL이 ASIL-B라면 SG-01 FSR이 H-05 위험을 충분히 커버한다는 별도 논증이 없음.
- **심각도: Major**
- **조치:** Agent-Safety는 H-05의 E 파라미터를 E3 → E4로 재검토하거나, E3 유지 시 "특정 이벤트 조합"의 구체적 예시와 발생 확률 근거를 추가하여야 함. 또한 H-05 단독 및 H-07과의 복합 시나리오 심각도를 별도 분석하고, SG-01 FSR에서 H-05를 충분히 커버하는 명시적 FSR 항목이 있음을 추적 문서로 연결할 것.

**ISSUE-002: 클럭 이상 및 전원 이상 고장 모드 위험원 미식별**

- **설명:** ARCHITECTURE.md §5.2의 확장 FFI 분석 표에서 **"전원 이상 감지 (전압 범위 이탈) → Level 3"** 및 **"타이머/클럭 공유 위협"** 이 명시적으로 식별되었다. SAFE_STATE_DEFINITION.md §3.1 전이 조건 매트릭스에도 "전원 이상 감지 → Level 3 (MPU 설정 소실 위험)"이 안전 응답으로 정의되어 있다. 그러나 본 HARA의 §3 위험원 분석 및 §4 위험도 평가에는 이에 대응하는 독립 위험원(H-xx)이 존재하지 않음.
  - 전원 이상은 MPU 설정 레지스터 소실 → FFI 보호 붕괴 → 실질적으로 H-02와 동등한 결과를 초래할 수 있는 독립적 고장 모드임.
  - 클럭 이상(비정상적 클럭 속도)은 SafetyFunction 타이밍 보장(SG-03)을 직접 침해함.
  - SafetyFunction 전용 타이머가 QM에 의해 변조되는 경우는 H-02/SG-02로 다루어지나, 하드웨어 클럭 자체 이상은 별도 위험원임.
- **심각도: Major**
- **조치:** Agent-Safety는 다음 위험원을 추가 검토하여 HARA에 포함하거나 분석 범위에서 명시적으로 제외(exclusion rationale 기재)할 것:
  - H-09 (가칭): 전원 전압 이상으로 인한 MPU 설정 소실 및 FFI 보호 붕괴
  - H-10 (가칭): 시스템 클럭 이상으로 인한 SafetyFunction 타이밍 보장 실패
  - 제외 시 근거: "하드웨어 레벨 고장 (§1.4 분석 범위 외)"으로 명시하되, AoU에서 통합자 하드웨어 설계 책임으로 위임함을 문서화할 것.

**ISSUE-003: H-04(오탐 Safe State 전이) S2 근거에 Safe State 설계 순환 의존**

- **설명:** H-04의 심각도 S2 근거에서 "Safe State 설계(Level 1 Degraded → Level 2 Controlled Stop)에 따라 완전 정지가 아닌 제한 운행으로 전환될 수 있음"을 완화 요소로 사용하였다. 이는 안전 메커니즘의 정상 동작을 위험도 평가의 완화 인자로 사용하는 것으로, ISO 26262 Part 3 Cl.15.4.4의 위험도 평가 원칙에 위배될 수 있음.
  - ISO 26262 Part 3에서 S 파라미터는 안전 메커니즘이 없는 최악의 경우(worst-case without safety mechanism)를 기준으로 평가해야 함.
  - Safe State Level 1이 정상 동작한다는 가정 없이, 단순히 "갑작스러운 차량 기능 중단"의 최악 결과를 평가해야 함.
  - 고속 주행 중 갑작스러운 파워스티어링/전동 브레이크 보조 중단은 운전자 적응 시간이 매우 짧으며, 후방 추돌 및 차선 이탈로 S3에 해당할 수 있는 시나리오를 배제할 수 없음.
  - S2 유지 시: 구체적인 차량 기능 예시(예: 정보 디스플레이 OFF 등 S2 수준의 영향에 그치는 기능)와 함께 근거를 보강해야 함.
- **심각도: Major**
- **조치:** Agent-Safety는 H-04 S 파라미터를 안전 메커니즘 작동을 가정하지 않은 최악 시나리오로 재평가할 것. S3으로 재평가 시 E4/C2 조합에서 **ASIL-C**가 되며, 이 경우 SG-04 ASIL 등급도 상향 조정 필요. S2 유지 시 Safe State 설계에 의존하지 않는 독립적 근거를 명시할 것.

**ISSUE-004: SG-03 [X]ms — 기능 안전 요구사항으로서의 미결 파라미터 처리 방식 부적절**

- **설명:** SG-03은 "위험 조건 감지 후 [X]ms 이내에 Safe State 전이"를 규정하나, [X]가 미결 상태임. §5.1에서 참조값 10ms를 제시하고 AoU-03으로 위임하는 방식을 채택하였으나, 이는 다음 문제를 내포함:
  - ASIL-D 안전 목표의 핵심 파라미터가 수치화되지 않은 채로 검증 기준이 될 수 없음. FSR-03-01, TSR, 검증 계획에서 "[X]ms"로 표기된 미결 파라미터는 합격 기준(acceptance criterion)을 정의할 수 없음.
  - 참조값 10ms가 제시되었으나, 이것이 HARA의 공식 파라미터인지 예시인지 불분명함. FSC의 FSR-03-01에서도 "[X]ms"로 그대로 사용됨.
  - ISO 26262 Part 3 Cl.15에서 안전 목표는 검증 가능(verifiable)해야 하며, 수치 미결 파라미터는 이 요건을 충족하지 못함.
  - SEooC에서 미결 파라미터 위임 자체는 허용되나, 범위(range)와 최악 경우(worst-case bound)를 HARA 단계에서 제시해야 함. 예: "[X]ms — 통합자 결정, OpenSafetyRTOS가 보장할 수 있는 최소값: Y ms".
- **심각도: Major**
- **조치:** Agent-Safety는 SG-03에 다음을 추가할 것: (1) OpenSafetyRTOS가 구현 가능한 최소 응답 시간의 상한(worst-case bound, WCET 기반)을 §5.1 또는 미결 사항 테이블에 명시, (2) AoU-03에서 통합자가 제시해야 하는 [X]ms의 허용 범위(예: 10ms ≤ X ≤ FTTI)를 HARA에 명기, (3) FSR-03-01 및 검증 계획에서 10ms를 잠정 기준값(provisional bound)으로 사용함을 명시.

**ISSUE-005: H-05의 안전 목표 연결 비명시적 — 추적성 갭**

- **설명:** §5.2 매핑 및 §7.1 추적성 표에서 H-05는 "SG-01에 의해 부분적으로 포괄됨"으로 표기되고 SG 열이 "—"(미할당)로 표기되어 있음. 그러나 ISO 26262 Part 3 Cl.15에서 식별된 모든 위험원은 안전 목표로 완전히 커버되어야 하며, "부분적 포괄"은 인증 관점에서 갭으로 지적될 수 있음.
  - H-05(ASIL-B)가 SG-01(ASIL-D) FSR에서 처리된다면, SG-01의 FSR 중 H-05를 명시적으로 커버하는 FSR ID를 추적성 표에 기재해야 함.
  - 또는 H-05에 대한 별도 안전 목표(SG-07: Watchdog 기반 복구 목표)를 도출하고 ASIL-B로 할당해야 함.
- **심각도: Minor**
- **조치:** Agent-Safety는 §7.1 추적성 표의 H-05 행에 커버링 FSR(FSR-01-01, FSR-01-03 등)을 명시하거나, 별도 SG-07을 도출하여 H-05 → SG-07 추적성을 완성할 것.

**ISSUE-006: OS-04(긴급 상황) 운영 상황의 노출도 처리 일관성**

- **설명:** OS-04(긴급 상황: ABS/ESC 활성, 긴급 제동 등)는 실제 차량 운행에서 발생 빈도가 낮음(E2~E3 수준). 그러나 H-01, H-02, H-03, H-05, H-08의 최악 경우 운영 상황으로 OS-04가 지정되면서도, 모두 E3~E4를 할당하고 있음.
  - H-01(E4), H-03(E4)에서 OS-04를 최악 운영 상황으로 지정하면서 E4(높음)를 할당한 것은, OS-04 자체의 노출도와 OS-01(정상 주행)의 노출도를 혼용한 것으로 보임.
  - OS-04의 발생 자체가 E2~E3인 상황에서 SafetyFunction 고장과의 동시 발생 확률은 더 낮을 수 있음. 단, SafetyFunction은 OS-04 이전에 이미 동작 중이므로 OS-01의 E4가 더 정확한 노출도임.
  - 이 혼동이 의도적이라면(즉, E4는 OS-01 기준), 최악 운영 상황 열에 OS-04를 지정한 것과 모순이 생김.
- **심각도: Minor**
- **조치:** Agent-Safety는 각 위험원의 "최악 운영 상황"과 "노출도 근거"가 일관되게 연결되도록 검토할 것. SafetyFunction이 지속 동작하는 상황에 대한 노출도(E4)는 OS-01 기준임을 명시하고, OS-04는 "심각도를 최대화하는 운영 상황"으로 구분하여 기술하는 것을 권고.

### 📋 권고사항 (Recommendations)

**REC-001: 복합 고장 시나리오(Combination of Faults) 분석 추가**
H-05(Watchdog 실패)와 H-07(부팅 자가진단 미감지)의 연쇄 시나리오, H-02(FFI 실패)와 H-06(Mailbox 오검증)의 동시 발생 시나리오는 ASIL-D 요구사항에서 잠재적으로 중요한 복합 고장 모드임. ISO 26262 Part 10 (Guidelines)에서 권고하는 복합 고장 고려를 FMEA 단계에서 반영하도록 HARA에 메모를 추가할 것.

**REC-002: Safe State 전이 완료 확인(Confirmation) 위험원 추가 검토**
현재 H-03은 "Safe State 전이 미수행"을 다루나, "Safe State 전이가 개시되었으나 완료되지 않은" 시나리오(예: Level 2 진입 시퀀스 중 중단)는 별도로 다루어지지 않음. SAFE_STATE_DEFINITION.md §4.2에서 Level 2 진입 시 "MPU 설정 재검증 실패 시 Level 3 에스컬레이션"이 정의되어 있으나, HARA 위험원 목록에 이 실패 모드가 없음. FMEA 단계에서 보완하거나, H-03의 설명 범위를 확장하도록 권고.

**REC-003: ASIL 결정 매트릭스 출처 표기 개선**
§4.2 표는 ISO 26262 Part 3 Table 4를 충실히 재현하였으나, 원문 Table 번호("Table 4")와 연도(2018년 3판)를 각주나 캡션에 명시하면 ISA 검토 시 추적 편의성이 향상됨.

**REC-004: AoU 문서(OSR-AOU-001) 현황 명시**
§6.3에서 AoU 항목 5개가 참조되나, OSR-AOU-001 문서의 현재 상태(Draft/미작성)가 명시되지 않음. HARA 내에 AoU 문서 상태를 기재하거나 SAFETY_CASE.md Open Points에 OP 항목을 추가할 것.
