## 2. 이전 이슈 조치 확인

### ✅ FSC-ISSUE-001 조치 확인 — 충족

**원래 지적:** FSR-01-04 부재 — Watchdog 타임아웃이 SafetyFunction 데드락 감지에 충분한지 보장하는 FSR 없음.

**v1.1 반영 내용:**
FSR-01-04가 다음과 같이 신설됨:

> "SafetyFunction은 Watchdog kick 주기를 SafetyFunction 태스크 실행 주기의 50% 이내로 설정해야 하며, Watchdog 타임아웃은 SafetyFunction 태스크의 WCRT × 3 이상이어야 한다."

검토자 평가:
- Kick 주기 ≤ 실행 주기 × 50%: Nyquist 기준 유사 마진 확보. 적절.
- Watchdog 타임아웃 ≥ WCRT × 3: 단일 WCRT 초과 후에도 2회 여유. 적절.
- 근거: "데드락 발생 시 Watchdog이 충분히 빠르게 감지하여 허용 가능한 제어 불능 시간 이내에 MCU 리셋이 발생하도록 보장"
- §4 FSR 요약 테이블 반영 확인
- ASIL-D 지정 확인

**판정: CLOSED — 조치 충족**

---

### ✅ FSC-ISSUE-002 조치 확인 — 충족

**원래 지적:** FSR-04-02가 기능 수준 알고리즘 프레임워크 없이 완전히 TSC로 위임. FSR로서 검증 불가.

**v1.1 반영 내용:**
FSR-04-02가 다음과 같이 구체화됨:

> - **일시적 오류**: 동일 Mailbox 슬롯에서 N회 이내 연속 오류 (N은 AoU로 정의, 기본값 N=3)
> - **영구적 오류**: N회 초과 연속 오류 또는 타임스탬프 기반 오류 지속 시간 > T_max
> - **일시적 오류 응답**: Safe State Level 1 진입 후 재시도
> - **영구적 오류 응답**: Safe State Level 2 또는 Level 3 전이
> - T_max 및 N의 구체적 값은 TSC에서 결정한다.

검토자 평가:
- 기능 수준 알고리즘 프레임워크 확립: N=3 기본값, T_max 개념 포함. 적절.
- "연속 N회" 판단 기준(동일 슬롯)이 명시됨. 검증 가능한 FSR로 격상됨.
- 파라미터(T_max 구체값, N 최종값)만 TSC로 위임: OSR-CR-FSC-001 ISSUE-002의 요구사항 정확히 충족.
- ASIL-D 상향 확인 (SG-04 HARA 재평가 연동)
- §6 미결 사항 표 항목 5번 업데이트 확인: "기본값 N=3; 알고리즘 프레임워크는 FSR-04-02에 정의됨. 파라미터만 TSC 위임"

**판정: CLOSED — 조치 충족**

---

### ✅ FSC-ISSUE-003 조치 확인 — 충족

**원래 지적:** 연속 검증 실패 시 Safe State 에스컬레이션 FSR(FSR-06-04) 부재. SAFE_STATE_DEFINITION.md와 불일치.

**v1.1 반영 내용:**
FSR-06-04가 다음과 같이 신설됨:

> "SafetyFunction은 동일 Mailbox 슬롯에서 연속 N회(기본값 N=3) 검증 실패 시 Safe State Level 2로 전이해야 한다. 단일 검증 실패는 Safe State Level 1 진입 후 재시도를 허용한다."

검토자 평가:
- 단일 실패 → Level 1 (재시도 허용): SG-04(오탐 최소화)와의 균형 유지. 적절.
- 연속 N회(기본 N=3) 실패 → Level 2: 에스컬레이션 로직 확립. 적절.
- SAFE_STATE_DEFINITION.md §3.1 전이 조건과 정합성 확인: "Mailbox CRC 검증 실패 (연속 N회) → Level 2" 일치.
- SG-06 커버리지 완성 확인
- ASIL-D 지정 확인

**판정: CLOSED — 조치 충족**

### ✅ FSC-ISSUE-004 조치 확인 — 조건부 충족

**원래 지적:** FSR-03-03 인터럽트 불가 구간 최대 허용 시간 미정. FSR-01-01과의 잠재적 상충(Watchdog kick 차단).

**v1.1 반영 내용:**
FSR-03-03에 다음 제약 추가:

> "Safe State 전이 함수는 인터럽트 불가(critical section) 구간에서 실행되어야 하며, 이 구간의 최대 실행 시간은 HW Watchdog 타임아웃의 10% 이내이어야 한다."

검토자 평가:
- 10% 제약 추가 확인. FSR-01-01과의 잠재적 상충 해소 의도 명확.
- OSR-CR-FSC-001 ISSUE-004 원래 요구: "50% 이내" 권고 → v1.1에서 더 엄격한 "10% 이내"로 구현. 더 보수적이므로 수용.
- 단, **10% 제약의 검증 방법 미명시**: §6 미결 사항 표 항목 6번에 "WCET 분석 도구(예: AbsInt aiT) 사용 권고"가 추가되었음을 확인.
- FSR-03-03과 FSR-01-04(WCRT × 3 타임아웃)의 수치적 정합성 검토:
  - FSR-01-04: Watchdog 타임아웃 ≥ WCRT × 3
  - FSR-03-03: 인터럽트 불가 구간 ≤ Watchdog 타임아웃 × 10%
  - 따라서: 인터럽트 불가 구간 ≤ (WCRT × 3) × 10% = WCRT × 0.3
  - WCRT 10ms 기준: 인터럽트 불가 구간 ≤ 3ms
  - 이 제약은 실현 가능하며 FSR-01-01과의 상충을 효과적으로 방지함.

**판정: CLOSED — 조치 충족**

## 3. 신규 중점 검토: SG-07 및 SG-08의 FSC 커버리지

### 🔴 FSC-NI-001 [CRITICAL]: SG-07 및 SG-08에 대한 FSR이 FSC에 완전히 부재

**설명:**

HARA v1.1(OSR-HARA-001)에서 다음 두 안전 목표가 신규 도출되었다:

| SG-ID | 내용 | 근거 위험원 | ASIL |
|-------|------|-----------|------|
| SG-07 | SafetyFunction 전용 클럭 소스는 QM 또는 외부 간섭으로부터 보호되어야 한다 | H-09 | ASIL-C |
| SG-08 | 전원 이상 시 SafetyFunction은 MPU 재설정을 수행하거나 Safe State Level 3으로 전이해야 한다 | H-10 | ASIL-C |

FSC v1.1(OSR-FSC-001)에서 SG-07 및 SG-08에 대한 커버리지를 검색한 결과:

- "SG-07" 검색 결과: **0건** (FSC 전체 문서)
- "SG-08" 검색 결과: **0건** (FSC 전체 문서)
- "FSR-07" 검색 결과: **0건**
- "FSR-08" 검색 결과: **0건**
- §3(SG별 FSR 분해) 절 목록: SG-01(§3.1), SG-02(§3.2), SG-03(§3.3), SG-04(§3.4), SG-05(§3.5), SG-06(§3.6) — SG-07 절, SG-08 절 **없음**
- §4 FSR 요약 테이블: FSR-01-xx ~ FSR-06-xx만 존재. SG-07, SG-08 행 **없음**
- §7 Agent-QA 확인 검토 요청: "모든 SG(SG-01~06)가 FSR로 완전히 분해되었는지" — SG-07, SG-08 **미언급**

**결론:** FSC v1.1은 HARA v1.1에서 도출된 SG-07과 SG-08을 전혀 반영하지 않았다. 이는 HARA → FSC 간 **완전한 추적성 단절**이다.

**위험 분석:**

*SG-07 (클럭 보호) 미커버 시 위험:*
- H-09 위험원(클럭 이상 → 타이밍 감시 실패)에 대한 기능 수준 대응 요구사항이 없음
- SafetyFunction 클럭 소스 이상을 감지하고 대응하는 기능이 FSC에 정의되지 않으면, 구현 단계(TSC, 코드)에서 이 기능이 누락될 가능성이 높음
- ASIL-C 위험원(H-09)이 FSR로 분해되지 않은 상태에서의 개발은 ISO 26262 Part 3 Cl.8 추적성 요건 위반

*SG-08 (전원 이상 MPU 복구) 미커버 시 위험:*
- H-10 위험원(전원 이상 → MPU 설정 소실 → FFI 붕괴)에 대한 감지 및 복구 요구사항 없음
- 전원 이상 시 MPU 설정 재검증 기능이 FSR로 정의되지 않으면 SafetyFunction이 FFI 붕괴 상태를 인지하지 못하고 계속 동작할 위험
- 이 위험은 H-02(FFI 실패, ASIL-D)와 동등한 결과를 초래할 수 있음 — ASIL-C인 H-10의 미커버는 사실상 ASIL-D 수준 위험을 야기

**심각도: Major — 즉각 조치 필요**

**필요 조치:**
Agent-Safety는 FSC에 다음 절을 추가해야 한다:

*§3.7 SG-07: SafetyFunction 전용 클럭 소스 보호 (ASIL-C) — 필요 FSR 예시:*
- FSR-07-01: SafetyFunction 전용 타이머 클럭 소스는 QM 파티션이 변경할 수 없어야 한다 (ASIL-C, 개발 ASIL-D — FSR-02-03과 연계하되 H/W 클럭 이상 감지로 확장)
- FSR-07-02: SafetyFunction은 클럭 이상(주파수 이탈, 클럭 정지)을 감지하는 메커니즘을 가져야 한다 (ASIL-C, 개발 ASIL-D) — 예: PLL lock 상태 모니터링, 독립 RC 오실레이터 교차 검증
- FSR-07-03: 클럭 이상 감지 시 SafetyFunction은 Safe State Level 3(Emergency Reset)으로 전이해야 한다 (ASIL-C, 개발 ASIL-D)

*§3.8 SG-08: 전원 이상 시 MPU 재설정 또는 Safe State Level 3 전이 (ASIL-C) — 필요 FSR 예시:*
- FSR-08-01: SafetyFunction은 전원 이상(전압 강하, brown-out) 감지 기능을 가져야 한다 (ASIL-C, 개발 ASIL-D) — 하드웨어 brown-out detector 활용 또는 voltage supervisor
- FSR-08-02: 전원 이상 복구 후 SafetyFunction은 MPU 설정 레지스터의 무결성을 검증해야 한다 (ASIL-C, 개발 ASIL-D) — 예상값과 실제 레지스터값 비교
- FSR-08-03: MPU 설정 무결성 검증 실패 시 Safe State Level 3(Emergency Reset)으로 전이해야 한다 (ASIL-C, 개발 ASIL-D)
- FSR-08-04: 전원 이상 감지 직후 MPU 설정 재초기화를 수행하거나 Level 3으로 전이해야 한다 (ASIL-C, 개발 ASIL-D)

*§4 FSR 요약 테이블에 FSR-07-01~03, FSR-08-01~04 행 추가 필요*
*§7 검토 요청 절의 SG 범위를 "SG-01~08"로 갱신 필요*

### 4.1 ASIL 등급 적합성

|---------|------------|---------|---------|
| FSR-01-xx | D | D | ✅ |
| FSR-02-xx | D | D | ✅ |
| FSR-03-xx | D | D | ✅ |
| FSR-04-01 | D (SG-04 상향) | D | ✅ |
| FSR-04-02 | D (SG-04 상향) | D | ✅ |
| FSR-05-xx | C | C (개발 D) | ✅ |
| FSR-06-xx | D | D | ✅ |

### 4.2 FSR-01-04 수치 정합성 검증

- Kick 주기 ≤ 태스크 실행 주기 × 50%: 논리적 마진 설계 요건 — 검증 가능 ✅
- 타임아웃 ≥ WCRT × 3: 데드락 감지 충분성 보장 — 검증 가능 ✅
- FSR-03-03과의 교차 정합성: 인터럽트 불가 구간 ≤ 타임아웃 × 10% = WCRT × 0.3 — 논리적 일관성 ✅

### 4.3 FSR-04-02와 FSR-06-04의 N 파라미터 일관성

FSR-04-02(일시적/영구적 오류 구분 N=3)와 FSR-06-04(연속 N=3 검증 실패 → Level 2)에서 동일한 N=3 기본값이 사용된다. 그러나 두 FSR의 N 파라미터가 독립적으로 정의되어 있어, TSC에서 FSR-04-02의 N을 다르게 설정할 경우 FSR-06-04와 불일치가 발생할 수 있다.

**권고 (Minor):** §6 미결 사항 표에 "FSR-04-02와 FSR-06-04의 N 파라미터는 동일한 값으로 통일되어야 함"을 명시할 것.

## 6. 최종 판정

**판정: 조건부 승인 (CONDITIONAL APPROVAL)**

### 판정 근거

**긍정적 평가:**
OSR-CR-FSC-001에서 지적된 4건의 Major 이슈(FSC-ISSUE-001~004)는 모두 v1.1에서 실질적으로 해소되었다:
- FSC-ISSUE-001 (FSR-01-04 추가): 충족 — Watchdog 타이밍 요구사항 확립
- FSC-ISSUE-002 (FSR-04-02 알고리즘 구체화): 충족 — N=3 기본값, 프레임워크 확립
- FSC-ISSUE-003 (FSR-06-04 추가): 충족 — 연속 실패 에스컬레이션 확립
- FSC-ISSUE-004 (FSR-03-03 시간 제약): 충족 — 10% 이내 제약 추가

**핵심 우려:**
HARA v1.1에서 신규 도출된 SG-07(ASIL-C)과 SG-08(ASIL-C)에 대한 FSR이 FSC v1.1에 **전혀 존재하지 않는다**. 이는 HARA → FSC 추적성의 완전한 단절로, ISO 26262 Part 3 Cl.8에서 요구하는 모든 안전 목표에 대한 FSR 분해 요건에 위반된다.

이 갭은 v1.1 작성 시 HARA 개정(SG-07, SG-08 추가)과 FSC 개정이 완전히 동기화되지 않아 발생한 누락으로 판단된다. ASIL-C 안전 목표 2개가 아무런 기능 수준 요구사항 없이 방치된 것은 **구조적 트레이서빌리티 결함**이다.
